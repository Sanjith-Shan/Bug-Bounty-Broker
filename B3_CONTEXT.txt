B³ — Bug Bounty Broker
Claude Code Project Context & Architecture

1. WHAT THIS IS
B³ is a sovereign agent running on EigenCloud that acts as a neutral, TEE-sealed intermediary
for smart contract bug bounties. It lets security researchers prove a vulnerability exists and get
paid without revealing it first, eliminating the trust asymmetry that plagues every existing
bounty platform.
The agent will be deployed as an EigenCompute app, presented at Eigen Labs' Private Preview
Demo Day on May 12, 2026.

2. WHY THIS MATTERS — THE PROBLEM
The trust asymmetry in bug bounties

Today's flow: researcher reveals full vulnerability → company decides if it's real → company
decides severity → company decides payment. The company has full information advantage at
every step.

3. EIGENCLOUD PLATFORM — WHAT WE HAVE
EigenCompute (our primary tool)

Docker containers running in Intel TDX TEEs on Google Cloud Confidential Space.
Encrypted memory (AES-XTS-128), hardware attestation via TDX Quote.
Each app gets a deterministic Ethereum wallet via process.env.MNEMONIC (BIP-39,
derived from app ID, stable across restarts if image digest unchanged).
Secrets in .env encrypted client-side by KMS, decrypted only inside attested enclave.
Variables with _PUBLIC suffix are visible externally.
On-chain app registry: image digest → Ethereum address mapping, publicly auditable.

System components
1. API Server (FastAPI, Python)

Primary interface. Runs inside EigenCompute TEE.
Endpoints:

POST /bounties — Company registers a bounty program (target contract, chain, severity

tiers, max payout per tier)
POST /submit — Researcher submits encrypted PoC (encrypted to agent's public key)
GET /attestations/{id} — Public attestation with signed severity, no exploit details
POST /deposit/{attestation_id} — Company deposits bounty (x402 or direct USDC

transfer to agent wallet)
GET /report/{attestation_id} — Returns full report ONLY after deposit confirmed

(gated)
GET /verify/{attestation_id} — Public verification endpoint: returns signed data,

image digest, instructions to verify on-chain
GET /programs — List active bounty programs
GET /health — Health check

2. Exploit Verification Engine

The core differentiator. Runs inside TEE.
For smart contract bounties (v1 focus):
1. Receive encrypted PoC (Solidity test, Foundry script, or transaction sequence)
2. Fork target chain at latest block using local Anvil/Hardhat fork
3. Execute the PoC against the fork
4. Measure: did it drain funds? How much? What access did it gain?
5. Compute funds-at-risk amount
6. All execution happens inside TEE — PoC never leaves enclave until payment
For v1 simplification: focus exclusively on Ethereum/EVM smart contract vulnerabilities. The
PoC format is a Foundry test file that the agent runs via forge test --fork-url .
3. Severity Assessment (EigenAI)

After exploit verification:
1. Feed exploit results + contract code + impact analysis to EigenAI ( qwen3-32b-128k-bf16 )
2. Prompt includes CVSS 3.1 scoring rubric
3. EigenAI returns deterministic CVSS score + severity classification + reasoning
4. Response includes signature — anyone can replay this exact assessment
5. Store: {prompt_hash, response_hash, eigenai_signature, cvss_score, severity}

The agent produces a signed attestation:

5. Escrow & Payment

Company deposits USDC to the agent's deterministic wallet address
Agent monitors for deposit matching the required bounty amount for the severity tier
On confirmed deposit: decrypt and release full PoC report to company
Transfer bounty minus fee (5-10%) to researcher's specified address
Fee stays in agent wallet to pay for compute costs (self-sustainability)

Signing attestations
python

from eth_account import Account
from eth_account.messages import encode_defunct
import json, os
def sign_attestation(attestation_data: dict) -> str:
"""Sign attestation with TEE-sealed wallet."""
mnemonic = os.environ["MNEMONIC"]
account = Account.from_mnemonic(mnemonic)
canonical = json.dumps(attestation_data, sort_keys=True, separators=(',', ':'))
message = encode_defunct(text=canonical)
signed = account.sign_message(message)
return signed.signature.hex()

9. WHAT NOT TO BUILD
Don't build a full prediction market or oracle. This is a bug bounty broker.
Don't build a generic "personal agent." Eigen Labs explicitly said these are not useful.
Don't overcomplicate the escrow — direct USDC transfers to/from the agent wallet are
sufficient for demo. Full smart contract escrow is a v2 feature.
Don't try to support non-EVM chains in v1. Stick to Ethereum/Base/Polygon forks.
Don't claim "trustless" — the TEE trust model is Intel+Google+Eigen. Say "minimized
trust" or "verifiable."

Don't spend time on user auth — for demo, the agent processes submissions
permissionlessly.

10. SUCCESS CRITERIA
The demo wins if:
1. A real exploit runs inside the TEE and produces a verifiable attestation
2. The severity assessment is deterministic (replay the same EigenAI call → same result)
3. Payment triggers report release automatically
4. A judge can click "Verify" and see cryptographic proof on-screen
5. The whole flow completes in under 60 seconds during the pitch
